Sécurité Active Directory GPO Durée : 3 mois
1 / 2

Contexte & Objectifs

Suite à un audit de sécurité interne, plusieurs vulnérabilités ont été identifiées dans l'infrastructure Active Directory. Le projet visait à renforcer la posture de sécurité de l'entreprise, tant sur l'environnement on-premise qu'en cloud (Microsoft Azure).

🔒 Phase 1 – Sécurisation Active Directory

  • Analyse des vulnérabilités : Audit complet avec BloodHound et PingCastle pour identifier les chemins d'attaque
  • Durcissement des GPO : Restriction RDP, désactivation de protocoles obsolètes (SMBv1), politique PowerShell en mode ConstrainedLanguage
  • Hygiène Active Directory : Réorganisation des OU, révision des droits NTFS, correction de l'héritage de permissions
  • Gestion des comptes privilégiés : Séparation des comptes admin, mise en place de PAW (Privileged Access Workstation)
  • Audit & Journalisation : Configuration avancée des politiques d'audit, centralisation des logs avec SIEM
  • Suivi proactif : Scripts PowerShell pour traquer les comptes inactifs via lastLogonTimestamp

☁️ Phase 2 – Sécurisation Azure

  • Audit Microsoft 365 : Revue complète d'Azure AD et des licences M365
  • Authentification renforcée : Déploiement MFA pour 100% des utilisateurs, policies d'accès conditionnel par localisation
  • Amélioration du Secure Score : Passage de 45% à 82% en désactivant les protocoles legacy (POP3/IMAP), activation de la protection anti-phishing
  • Gouvernance : Mise en place de labels de sensibilité, DLP (Data Loss Prevention)
  • Documentation : Cartographie complète de l'environnement cloud, plan d'amélioration continue, runbooks de sécurité

Automatisation n8n Exchange Durée : 1 mois
1 / 2

Contexte & Objectifs

La gestion manuelle des délégations Exchange (accès aux boîtes partagées, permissions FullAccess/SendAs) était chronophage et source d'erreurs. Le projet visait à automatiser entièrement le processus via un workflow n8n, de la demande à l'application des droits.

🔧 Architecture du workflow

  • 1. Demande initiale : Formulaire web pour soumettre une demande de délégation (utilisateur, boîte cible, type de droit, durée)
  • 2. Validation hiérarchique : Notification automatique au manager pour approbation/refus via email avec boutons d'action
  • 3. Application des droits : Exécution de commandes PowerShell via Graph API ou Remote PS pour attribuer les permissions Exchange
  • 4. Notification utilisateur : Email de confirmation avec détails des accès accordés
  • 5. Révocation automatique : Cron job quotidien qui retire les droits expirés selon la date de fin définie
  • 6. Consultation des accès actifs : Interface web pour visualiser toutes les délégations en cours
  • 7. Rapport hebdomadaire : Synthèse envoyée au support IT avec stats (nouvelles demandes, révocations, erreurs)

💡 Innovations techniques

  • Gestion des erreurs robuste : Retry automatique, logs détaillés, notifications en cas d'échec
  • Traçabilité complète : Historique de toutes les opérations dans une base de données SQLite
  • Sécurité : Authentification OAuth2, chiffrement des credentials, principe du moindre privilège
  • Évolutivité : Architecture modulaire permettant d'ajouter facilement d'autres types de délégations (SharePoint, Teams…)

Infrastructure Active Directory Azure AD Connect Durée : 45 minutes
1 / 1

Contexte & Objectifs

Dans le cadre de la maintenance de l'infrastructure Active Directory du Groupe FSP (domaine ad.fsp-groupe.com), il a été décidé de basculer le rôle de contrôleur de domaine « principal » de SRV-DC-001 vers SRV-DC-002. L'objectif : transférer les 5 rôles FSMO, reconfigurer le service de temps (W32Time) et migrer l'instance active d'Azure AD Connect — sans interruption de service.

Note : Dans Active Directory, il n'existe pas de contrôleur « principal » ou « secondaire » au sens strict. Cette opération désigne SRV-DC-002 comme détenteur des rôles FSMO et serveur AAD Connect actif.

📋 Pré-requis & Sauvegardes

  • Vérification réplication AD : repadmin /replsummary — réplication OK entre les deux DC
  • Validation rôles FSMO : netdom query fsmo — 5 rôles sur SRV-DC-001
  • État Azure AD Connect : Get-ADSyncScheduler — actif sur SRV-DC-001
  • Sauvegardes : Snapshots VM des deux DC + export configuration AAD Connect

🔧 Phase 1 – Transfert des 5 rôles FSMO

  • Méthode : Transfert propre via ntdsutil (Transfer, non Seizure) car les deux DC sont en ligne
  • Rôles transférés : Schema Master, Domain Naming Master, PDC Emulator, RID Master, Infrastructure Master
  • Validation : netdom query fsmo — 5 rôles confirmés sur SRV-DC-002

⏱️ Phase 2 – Configuration du service de temps (W32Time)

  • SRV-DC-002 (nouveau PDC) : Configuré comme source NTP fiable (0.fr.pool.ntp.org, 1.fr.pool.ntp.org), /reliable:yes
  • SRV-DC-001 : Reconfiguré en /reliable:no, synchronisation sur SRV-DC-002
  • Validation : w32tm /query /source — chaîne de synchronisation correcte

☁️ Phase 3 – Bascule Azure AD Connect

  • Étape 1 : Vérification des versions AAD Connect identiques sur les deux serveurs
  • Étape 2 : Passage de SRV-DC-001 en Staging Mode (synchronisation arrêtée immédiatement)
  • Étape 3 : Désactivation du Staging Mode sur SRV-DC-002 (devient actif)
  • Étape 4 : Synchronisation initiale forcée (Start-ADSyncSyncCycle -PolicyType Initial)
  • Étape 5 : Validation dans le portail Entra ID — dernière synchronisation récente, aucune erreur

✅ Vérifications post-bascule

  • Script PowerShell de validation automatisée en 5 étapes : FSMO, réplication AD, service de temps, AAD Connect, événements critiques
  • Tests fonctionnels : connexion utilisateur, accès partage réseau, application GPO
  • Plan de retour arrière : Procédure de rollback documentée pour chaque phase (FSMO, W32Time, AAD Connect)

Contexte & Objectifs

Pour renforcer la gestion des mots de passe au sein du Groupe FSP, j'ai déployé Vaultwarden (implémentation open source et légère de Bitwarden) en interne. L'objectif : fournir un coffre-fort de mots de passe sécurisé, synchronisé avec Active Directory, et déployé automatiquement sur les postes via Microsoft Intune.

🔧 Architecture & Réalisations

  • Déploiement Docker : Stack Docker Compose avec Vaultwarden + conteneur de synchronisation LDAP, volumes persistants, politique de redémarrage automatique
  • Reverse proxy Traefik : Exposition via HTTPS avec certificat PKI interne (Step-CA / ACME), sur le domaine bitwarden.ad.fsp-groupe.com
  • Synchronisation LDAP : Conteneur vaultwarden_ldap connecté à Active Directory pour la création automatique des comptes utilisateurs à partir de l'annuaire
  • Sécurité : Inscriptions libres désactivées (invitation admin uniquement), accès réseau interne/VPN uniquement, token d'administration sécurisé
  • Déploiement extension Edge via Intune : Profil Settings Catalog pour l'installation silencieuse de l'extension Bitwarden + script PowerShell de préconfiguration du serveur interne
  • Administration : Interface d'administration web pour gérer les utilisateurs et les invitations

💡 Points techniques clés

  • Infrastructure as Code : Déploiement reproductible via docker-compose.yml
  • TLS bout en bout : Certificat PKI interne via Traefik, aucun port applicatif exposé
  • Intégration AD : Filtre LDAP ciblant uniquement les comptes actifs (userAccountControl)
  • Déploiement GPO + Intune : Extension Edge installée silencieusement, serveur préconfiguré via registre Windows
  • Documentation complète : Procédures de déploiement, configuration LDAP, administration, sauvegardes

Cybersécurité Détection d'incident FOVI / Whaling Forensique Détection & rapport : février 2026

Contexte & détection

En février 2026, j'ai détecté un cas d'usurpation d'identité externe visant l'entreprise. Un fraudeur s'est fait passer pour une collaboratrice (négociatrice) et a utilisé l'identité du Directeur Général (signature usurpée) pour faire exécuter un faux ordre de virement par un débiteur. Le débiteur a effectué un virement vers un compte personnel frauduleux, croyant régler une échéance auprès de la société.

🔍 Investigation technique

  • Audit complet : contrôle du compte email et du poste de la collaboratrice usurpée — aucune compromission détectée
  • Analyse des logs de messagerie : confirmation qu'aucun email frauduleux n'a quitté l'organisation
  • Analyse comparative : emails légitime vs frauduleux (domaine officiel vs ProtonMail), et RIB légitime vs faux RIB (titulaire, IBAN, signature)
  • Caractérisation du vecteur : attaque FOVI / Whaling par création d'une adresse ProtonMail imitant l'adresse légitime, avec faux RIB en pièce jointe
  • Conclusion : usurpation 100 % externe — aucun système interne compromis

📋 Actions menées

  • Alerte du débiteur : confirmation du RIB officiel, blocage de tout virement vers le compte frauduleux, conseil de dépôt de plainte
  • Recommandations sécurité : renouvellement des mots de passe côté débiteur pour couper tout accès résiduel
  • Communication interne : sensibilisation des collaborateurs aux techniques de phishing / FOVI
  • Rapport d'incident : rédaction d'un compte rendu complet (chronologie, vecteur d'attaque, preuves, recommandations) signé par le DSI
